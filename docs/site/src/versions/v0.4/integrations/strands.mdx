Strands models a conversation differently from a flat message list. Its message `role` is
only ever `user` or `assistant` — there is **no system role in the message list**. The
system prompt is a separate argument to the agent. Message content is also a list of
content blocks rather than a bare string, so a text turn is `{ text: "..." }`.

So the bridge **partitions** a Prompting Press composition into two parts:

- **System prompt** — every `system`-role text, joined in order with a blank line, becomes
  the single `system_prompt` / `systemPrompt` argument (or nothing, if there is no system
  message).
- **Messages** — every other turn maps to `{ role, content: [{ text }] }`.

<Aside type="caution" title="System position is flattened">
Strands has one system prompt, not a system message you can place mid-conversation. Every
`system`-role text is hoisted into that single prompt regardless of where it sat in your
composition. If your design depends on a system instruction appearing *between* turns,
that ordering can't be carried across — it is a Strands modelling choice, not something
Prompting Press can preserve.
</Aside>

## What's not mapped

The recipe maps plain text only. Strands' provider-specific content blocks — guard content,
tool-use and tool-result blocks, cache points, and the like — are request-assembly concerns
that belong to your call layer, not to a rendered prompt. Prompting Press produces text; if
you need those blocks, add them in your own Strands code around the mapped messages.
