## T032 ŌĆö per-construct classification table

Empirically probed against MiniJinja **2.21.0** (the pinned version) by parsing each fixture
body directly through `Environment::add_template_owned` under the kernel's locked feature set
(`macros` / `multi_template` disabled). Probe was temporary and has been removed.

| Construct (fixture)                         | `minijinja::ErrorKind` | MiniJinja detail string          | `KernelError` (after heuristic fix) | `KernelError` (before fix) |
|---------------------------------------------|------------------------|----------------------------------|-------------------------------------|----------------------------|
| `{% include "x" %}` (`excluded-include`)    | `SyntaxError`          | `unknown statement include`      | `ExcludedFeature`                   | `Parse`                    |
| `{% extends "base" %}` (`excluded-extends`) | `SyntaxError`          | `unknown statement extends`      | `ExcludedFeature`                   | `Parse`                    |
| `{% import "m" as m %}` (`excluded-import`) | `SyntaxError`          | `unknown statement import`       | `ExcludedFeature`                   | `Parse`                    |
| `{% from "m" import f %}` (`excluded-from-import`) | `SyntaxError`    | `unknown statement from`         | `ExcludedFeature`                   | `Parse`                    |
| `{% macro f() %}{% endmacro %}` (`excluded-macro`) | `SyntaxError`     | `unknown statement macro`        | `ExcludedFeature`                   | `Parse`                    |
| `{% block b %}{% endblock %}` (`excluded-block`)   | `SyntaxError`     | `unknown statement block`        | `ExcludedFeature`                   | `Parse`                    |

Key facts:

- All six produce the **same** `ErrorKind::SyntaxError` ŌĆö the kind alone does NOT distinguish
  an excluded feature from an ordinary syntax error (research D4 confirmed). The discriminator
  is the detail string only.
- The detail is reliably `unknown statement <kw>` with the keyword **bare** (not `'include'` /
  `` `include` ``).

## Heuristic change (did touch `engine.rs` ŌĆö justified)

The task authorized a cheap, correct improvement to `looks_like_excluded_feature`. I made one.

- **Before:** matched `'<kw>'` or `` `<kw>` `` (quoted/backticked). MiniJinja 2.21.0 emits the
  keyword bare, so the match **never fired** ŌåÆ every excluded feature fell back to `Parse`.
  SC-008 was still satisfied (Parse is an accepted rejection variant), but classification
  precision was 0/6.
- **After:** matches the phrase `unknown statement <kw>` for the six disabled-tag keywords
  (`include`, `extends`, `import`, `from`, `macro`, `block`). Now 6/6 classify as
  `ExcludedFeature`.
- **Why this is tight, not a loosening:**
  - Ordinary syntax errors (unclosed `{{`, bad token, etc.) do **not** emit the
    `unknown statement` phrase, so they remain `Parse` ŌĆö verified by a new fixture
    (`malformed-syntax.json`, body `Hello {{ name `) and two tests
    (`ordinary_syntax_error_is_parse_not_excluded_feature` integration test +
    `engine::tests::ordinary_syntax_error_detail_is_not_excluded` unit test).
  - It keys off the full `unknown statement <kw>` phrase, not a loose substring (so a stray
    `"block"` in an unrelated message cannot trip it).
  - A genuinely unknown tag that is not one of the six (e.g. `{% frobnicate %}`) emits
    `unknown statement frobnicate`; `frobnicate` is not in the keyword set, so it correctly
    falls through to `Parse` ŌĆö asserted in the unit test.
- The heuristic still only refines the *label*; both branches are loud parse-time errors, so a
  future MiniJinja wording change degrades gracefully to `Parse` (still SC-008-compliant) and
  is caught loudly by the precision early-warning test (see below).

## T033 ŌĆö rustdoc added

`src/lib.rs` (`//!` crate docs), rewritten to cover:

- **Four capabilities** ŌĆö render (strict-undefined; six excluded features rejected at parse);
  agreement analysis (returns required roots only, the ŌŖå check is the consumer's);
  variant resolution + provenance/hashing (`template_hash`/`render_hash`, no `vars_hash`);
  var-provenance + opt-in guard.
- **Invariants** ŌĆö FFI-free (Principle II / C-02), validation-blind (FR-004), no-I/O
  (Principle III / C-03), and **error normalization is the consumer's job, not the kernel's**
  (C-06 / Principle VI).
- **Normative "what the kernel does NOT do with what it returns" (X1 / SEC-002):**
  - the guard field does NOT sanitize; untrusted/external values pass through byte-for-byte and
    the rendered body is unchanged whether the guard is on or off (SC-005);
  - provenance tags are declarative metadata with NO runtime enforcement;
  - `output_model` is a reference that is never parsed.
- **Doctest example** ŌĆö deserializes a `PromptDefinition` from canonical JSON and renders it
  (`Hello {{ name }}` ŌåÆ `Hello Ada`), asserting variant `default`, a 64-char hex
  `template_hash`, and `guard.is_none()`. Compiles + runs under `cargo test --doc`.

Mirrored the no-sanitization / no-enforcement invariant onto the two types a consumer might
reach directly (`src/provenance.rs`):

- `GuardConfig` ŌĆö "This is NOT a sanitizer (X1/SEC-002)": only produces an advisory string
  naming fields; never inspects/escapes/rewrites a value; body byte-identical on/off.
- `ProvenanceView` ŌĆö "declarative metadata, NOT runtime enforcement": only reports tags; the
  kernel never gates/sanitizes/alters rendering based on provenance.
