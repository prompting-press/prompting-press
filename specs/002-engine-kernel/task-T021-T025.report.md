## How the allowlist is derived from the env (FR-020) — the exact minijinja API

**Accessor used:** `minijinja::Environment::globals(&self) -> impl Iterator<Item = (&str, Value)>`.

Verified directly against the installed pinned source
(`~/.cargo/registry/src/index.crates.io-*/minijinja-2.21.0/src/environment.rs:786`):

```rust
/// Returns an iterator of all global variables.
pub fn globals(&self) -> impl Iterator<Item = (&str, Value)> {
    self.globals.iter().map(|(key, value)| (key as &str, value.clone()))
}
```

This is a **stable public method** — NOT behind `unstable_machinery`, NOT feature-gated.
The kernel builds its canonical env (`engine::build_environment`), enumerates
`env.globals()`, collects the names into a `BTreeSet<String>`, and subtracts that set from
`undeclared_variables(false)`:

```rust
fn env_globals(env: &minijinja::Environment<'_>) -> BTreeSet<String> {
    env.globals().map(|(name, _value)| name.to_string()).collect()
}
```

**This is NOT a hardcoded list.** It is read from the live `Environment` the kernel renders
against, so it cannot drift from the actual engine config (re-enabling a feature that
registers a new global, or an `add_global` call, is reflected automatically). Under the
kernel feature set (`builtins` on) this yields exactly `range`, `dict`, `debug`, `namespace`
— verified in `minijinja-2.21.0/src/defaults.rs::build_globals`, all four inside the
`#[cfg(feature = "builtins")]` block. (Note: `debug` is registered as a *global function* by
`builtins` even though the `debug` *feature* is off — deriving from the env catches this
correctly where a hardcoded `["range","dict","namespace"]` would have been wrong.)

Built-in **filters and tests are NOT reported** by `undeclared_variables` (they are
syntactically distinct from variable lookups), so they need no allowlist entry. Documented in
`agreement.rs`.

## How the parse-error short-circuit works (FR-016a / T025)

`undeclared_variables` returns an **empty set on parse failure** — verified in
`minijinja-2.21.0/src/template.rs:425–436`:

```rust
pub fn undeclared_variables(&self, nested: bool) -> HashSet<String> {
    match parse(...) {
        Ok(ast) => find_undeclared(&ast, nested),
        Err(_) => HashSet::new(),   // <-- the footgun
    }
}
```

The kernel forecloses the footgun by **parsing the template FIRST**: it calls
`env.add_template_owned(...)` (which parses eagerly) and `env.get_template(...)`, mapping any
error via `engine::map_minijinja_error` to `Err(ExcludedFeature | Parse)` **before** it ever
reaches `undeclared_variables`. With `macros` / `multi_template` disabled (research D1/D4),
an excluded-feature tag (`{% include %}` etc.) is an unrecognised tag and fails at this parse
step too. A non-parseable source therefore never yields a `Template` handle, making the
empty-set branch **structurally unreachable** from `required_roots`. A broken or
excluded-feature template can never masquerade as "requires no variables." Proven by test
`v4_3_excluded_feature_template_errors_never_empty_agreement` (asserts `Err`, not an empty
`Agreement`).

## Soundness notes (what the engine guarantees vs. what the kernel adds)

- **Engine guarantees (NOT re-filtered by the kernel):** `undeclared_variables(false)`
  returns ROOT names only (`user`, not `user.name`) and excludes loop locals, `{% set %}`
  targets, and block/with locals. The kernel does not reimplement this filtering; tests V2.1
  (nested field excluded), V2.2 (loop local excluded), V2.3 (set target excluded) verify the
  engine's behaviour through the kernel.
- **Kernel adds:** globals subtraction (V2.4 — `range` and `namespace` both verified absent
  while the real root `n` is present) and the parse-error short-circuit (V4.3).

## V2.5 purity approach

`required_roots(&PromptDefinition, ...)` takes a **shared borrow** of the def (cannot mutate
through the type system) and **never takes `values`** (it does not render). The test
additionally: snapshots the def via `serde_json::to_string` before/after and asserts equal;
holds a `values` map + clone and asserts equal after (the analysis never reads them); and
asserts `Agreement` carries only the roots set (no `text` field exists to leak a side-effect
render). FR-018 / SC-006.
