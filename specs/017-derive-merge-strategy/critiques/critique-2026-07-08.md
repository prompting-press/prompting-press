# Critique Report: 017 — derive() merge strategy

**Date**: 2026-07-08 | **Reviewed**: spec.md, plan.md, tasks.md, contracts/, data-model.md,
research.md, constitution v2.0.0, DECISIONS.md

## Executive Summary

The spec/plan/tasks are **solid and implementation-ready**. The feature is tightly scoped
(consumer/binding-layer, no kernel/schema/I/O), the risky decisions were resolved during grill +
clarify, and the constitution amendment is correctly scoped. **Verdict: ⚠️ PROCEED WITH UPDATES** —
no blocking flaws, but four findings worth folding in before implementation (one is a genuine
soundness question the plan under-specifies; the rest are recommendations/questions). None require
re-running specify/plan.

## Product Lens Findings

### 3a. Problem Validation
- **Clear.** Real consumer (Bellwether trend_value: shared `extraction` base + per-child adds).
  The problem — "inherit + add variables without dropping the base's" — is concrete and evidenced.

### 3b. User Value / MVP
- **P1 (💡):** The spec unions ALL THREE map fields (`variables`, `variants`, `metadata`) but the
  *only* evidenced consumer need is `variables`. `variants`/`metadata` union is "consistency-driven"
  (spec Assumptions + memory Q4). This is defensible (one rule is simpler than a per-field split) but
  it ships **unproven surface** — mild tension with C-08 (earn generality). *Suggestion:* keep the
  uniform rule (churn of special-casing isn't worth it) but explicitly document `variables` as the
  primary/tested use and add at least one `variants` + one `metadata` union test so the surface isn't
  merely asserted. (Tasks T010/T015/T016 currently emphasize `variables` — extend them.)

### 3c. Alternative Approaches
- **Clear.** The simpler alternative (manual spread with today's `derive`) is documented as US3's
  baseline; the strategy removes a real footgun (silently dropping inherited vars). YAML `extends`
  and Jinja inheritance were considered and rejected with rationale.

### 3d. Edge Cases
- **Strong.** Empty-map, absent-field, key-collision, agreement-violation, coverage-miss, unknown-
  strategy all enumerated in spec Edge Cases + quickstart scenarios.

### 3e. Success Measurement
- **Clear.** SC-001..SC-007 are measurable (parity tests, no-diff assertions, amendment recorded).

## Engineering Lens Findings

### 4a. Architecture Soundness
- **Strong.** Merge logic lands once in the Rust consumer; bindings delegate (verified). Matches
  Principle I. No new coupling.

### 4b/4d. Failure Modes / Performance
- **N/A-clean.** No I/O, no concurrency, no scaling axis. Merge is a bounded in-memory map op at
  construction. Determinism is the only relevant property and it's covered (SC-003 parity + hashes).

### 4c. Security & Privacy
- **E1 (💡):** `metadata` union means a child can now inherit a base's opaque metadata entries. If a
  base carries a `guard` key or model hints in metadata, `Shallow` silently propagates them to the
  child. Not a vulnerability (metadata is opaque/echoed, no behavior), but worth a doc note so
  consumers aren't surprised that metadata is inherited under `Shallow`. *Suggestion:* note it in the
  T024 docs task.

### 4e. Testing Strategy
- **E2 (🎯 — the one real gap):** The plan asserts "re-validate the merged whole via `Prompt::new`"
  is sufficient, but does **not** enumerate the cross-field interaction under `Shallow`: a merged
  **variant body** may reference a variable that survives only because of how `variables` merged. The
  agreement check must run over **every arm** (root + all variants) against the **merged** variable
  set. `Prompt::new` already does per-arm `required_roots` (prompt.rs invariant 2/3), so this is
  likely handled — but it is **not tested**. *Suggestion:* add an explicit test: base declares
  `extraction`, base has a variant `terse` whose body uses `{{ extraction }}`; child `Shallow`-merges
  a `variables` overlay that (via `Replace` semantics on a *different* field) could drop `extraction`
  — assert the merged variant either validates or fails correctly. Add to T010. This closes the
  "passes construction but semantically broken" risk the adversary was asked to probe.

### 4f. Operational Readiness
- **Clear.** Library feature; rollback = revert. No migration. The amendment is the only stateful
  artifact and is reversible.

### 4g. Dependencies
- **Clear.** Zero new dependencies (SC-006). Confirmed in plan Technical Context.

## Cross-Lens Insights

- **X1 (💡):** The TS breaking change (moving `validators` into an options object, research R4/T014)
  converges product + engineering: it's the correct C-11 shape *and* a consumer-visible break. At 0.x
  it's acceptable, but it should be called out in the CHANGELOG/migration note, not just the spec.
  *Suggestion:* add a migration line to T024 docs — "TS `derive(overlay, validators)` →
  `derive(overlay, { validators, merge })`". Check whether any in-repo example/sample or the
  Composition API calls `derive(overlay, validators)` positionally and would break (T014 should grep
  for call sites).
