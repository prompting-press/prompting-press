# Critique Report: 018 — provenance attributes helper

**Date**: 2026-07-08 | **Reviewed**: spec.md, plan.md, tasks.md, contracts, data-model, research,
constitution v2.0.0 (→ v3.0.0), DECISIONS.md

## Executive Summary

Tight, low-risk, implementation-ready. A pure projection of four existing fields; the risky
alternative (the callback `ProvenanceSink`/`OtelSink`) was correctly rejected. **Verdict: ⚠️ PROCEED
WITH UPDATES** — no blocking issues; three items to fold in (adoption-fit, parity mechanism, and the
OTel-namespace honesty), all task/spec-level.

## Product Lens

- **3a Problem** — Clear: removes hand-rolled `span.set_attribute` boilerplate; the field-name/key
  mapping is the tedious part the helper owns.
- **3b Value / P1 (💡 — adoption risk):** the helper emits only 4 fields and NO metadata. The
  challenger question worth pre-empting: is a 4-field helper too thin, so consumers who want
  `model_hint`/`experiment` tags hand-roll anyway and ignore it? *Mitigation:* the spec's answer
  (read the 4 public fields for custom keys; metadata excluded on purpose) is defensible, but the
  docs (T019) should show the "helper + your own extra attrs" pattern so adopters see the helper
  covers provenance and their metadata is one more line — not an either/or.
- **3c Alternatives** — Clear: sink rejected (Principle V); helper is the minimal formatting layer.
- **3e Success** — Measurable (parity, no-dep, exclusion, exact-4).

## Engineering Lens

- **4a Architecture (💡):** "keys defined once in the consumer, mirrored by each binding" — each
  binding surfaces its OWN `RenderResult` (py pyclass, napi struct, TS type). The plan should be
  explicit about whether Python/TS truly reach a shared Rust constant or re-declare the 4 key
  strings. If re-declared, that's a 3-place parity risk. *Suggestion:* make the conformance case
  (T010) the enforcer, and add a source comment cross-referencing the canonical constants.
- **4c Security** — covered by the security review (SEC-001 exclusion-by-test; SEC-002 hash-is-
  identifier note). No new surface.
- **4d Parity / D1 (🤔):** the four values are already strings; is D1 (date/decimal serialization)
  even relevant, making T010 partly busywork? Likely the values are pre-serialized, so the parity
  case is cheap insurance rather than catching a real marshaling delta — keep it, but don't over-
  invest; the real guard is "exact 4 keys + identical values."

## Cross-Lens

- **X1 (💡 — honesty):** `template_hash`/`render_hash` are NOT in the OTel GenAI convention; emitting
  them under `gen_ai.prompt.*` could mislead a backend that expects convention compliance. The
  clarify default (keep `gen_ai.prompt.*` + doc note) is reasonable, but the challenger may argue a
  library-owned namespace is safer. Hold pending the challenger; the doc note (FR-003) is the
  minimum.

## Verdict: ⚠️ PROCEED WITH UPDATES

No Must-Address. Recommendations are doc/spec/task-level. Hold X1 for the running challenger, then
fold all via one iterate pass.
