## Message / Composition API

```rust
// crates/prompting-press/src/compose.rs (re-exported at crate root: Composition, Message)

pub struct Message {
    pub role: String, // prompt def's role stringified: "system" | "user" | "assistant"
    pub text: String, // the entry's rendered body
}

impl Composition {
    pub fn new() -> Self;                       // empty; resolves to Ok(vec![]) (F7)
    pub fn is_empty(&self) -> bool;
    pub fn len(&self) -> usize;

    // Fallible, &mut self, returns Result<(), ConsumerError> — NOT Self (not chainable).
    pub fn append<V>(&mut self, name: &str, vars: &V, variant: Option<&str>)
        -> Result<(), ConsumerError>
    where V: Serialize + Validate, V::Context: Default;

    pub fn resolve(&self, reg: &Registry) -> Result<Vec<Message>, ConsumerError>;
}
```

- Ordered builder: `new()` + `append(...)`. **No `.chain()`** (FR-013) — it would collide with
  `Iterator::chain` and can't cross PyO3/napi. `append` returns `Result<(), _>`, not `Self`, so
  it cannot be fluently chained even by accident.
- `Message.role` is `def.role.to_string()` via the generated `PromptDefinitionRole: Display`
  (`crates/prompting-press-core/src/generated/prompt_definition.rs:250`).

## Append-vars typing: chose option (a)

**Option (a): validate + serialize eagerly at `append`.** Each entry carries a different typed
`V: Serialize + Validate`, but all entries live in one homogeneous `Vec`. `append` runs
`vars.validate()` once, then `minijinja::Value::from_serialize(vars)`, and stores
`Entry { name, values: minijinja::Value, variant: Option<String> }`. `resolve` then only
resolves each prompt by name and renders the stored, already-validated value.

Why (a) over (b) (boxed trait objects deferring validation to `resolve`):

- **No `dyn` gymnastics.** The only erased thing is the already-validated `minijinja::Value` —
  the exact type the kernel renders against anyway. No boxed closures, no trait-object lifetime
  juggling.
- **Reuses the same validate→serialize bridge as `crate::render`** (`render.rs` lines 83–87):
  `vars.validate().map_err(ConsumerError::from)?` then `Value::from_serialize(vars)`. No
  duplicated validation/serialization logic, no duplicated render chain (FR-011 / C-01 — render
  itself is `prompting_press_core::render`).
- **Keeps both invariants intact.** Validation always runs (eagerly, at append). No partial
  result is ever returned as success: an invalid entry fails *at `append`* and is never stored,
  so `resolve` only ever sees fully-validated entries. A `resolve`-time failure (unknown prompt
  / unknown variant / strict-undefined) propagates via `?` and discards the partial `Vec`.

Trade-off vs the spec's "validate at render" wording: under (a) validation is at *append*, not
*resolve*. This is documented in the module rustdoc. It is still strictly "validation happens,
no entry renders unvalidated, no partial-as-success" — the whole composition is validated
incrementally as built. (b) would move the same checks to `resolve`; either satisfies the US4
contract. (a) is the cleaner-compiling choice.

## Where validation happens

- **Vars validation:** eagerly in `append` (garde `validate()` once per entry, normalized to
  `ConsumerError::Validation`). Fail-fast: bad entry not stored.
- **Prompt resolution + render errors:** at `resolve` — `reg.get(name)` (absent →
  `ConsumerError::UnknownPrompt`) and `prompting_press_core::render` (kernel errors →
  `ConsumerError::Kernel`, with `Parse`/`Render` detail scrubbed by the existing `From` impl).
- **Composition uses no guard expansion:** `resolve` passes `&GuardConfig::default()`, which
  leaves `text` unchanged (guard text is never concatenated into `text`; spec 002).
