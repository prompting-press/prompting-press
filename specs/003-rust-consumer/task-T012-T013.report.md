## Loader signatures + chosen return-shape

```rust
impl Registry {
    pub fn load_json(&mut self, doc: &str) -> Result<&PromptDefinition, ConsumerError>;
    pub fn load_yaml(&mut self, doc: &str) -> Result<&PromptDefinition, ConsumerError>;

    // private shared tail:
    fn insert_and_get(&mut self, def: PromptDefinition) -> &PromptDefinition;
}
```

**Return-shape decision: `Result<&PromptDefinition, ConsumerError>` (return a ref to the
inserted def).** This is the contract's preferred shape (`contracts/consumer-api.md` §1) and
it compiles cleanly, so no fallback (`Result<(), _>` / returning the name) was needed.

The borrow-checker friction the task flagged (insert via `&mut self`, then hand back a `&`)
is resolved by the private `insert_and_get`: capture the owned `name` key, `insert`, then
re-`get(&key)` and return that immutable borrow. The re-`get` is a single `BTreeMap` lookup;
the `expect("just inserted under this key")` is infallible (we inserted under that exact key
on the line above). This keeps both public loaders one-liners over a single insert+borrow
path and avoids any `unsafe`/`unwrap`-on-None footgun.

**Both loaders normalize into the SAME `PromptDefinition`** (the kernel's re-exported,
`#[derive(Deserialize)]` shape) — `serde_json::from_str::<PromptDefinition>` and
`serde_yaml_ng::from_str::<PromptDefinition>` target the identical type. No parallel shape
(FR-005/006/008).

**Error mapping**: deserialize errors map to `ConsumerError::Load(e.to_string())`. As the
task and `error.rs` note, the serde error string is a *parse-location* description (line/col,
`missing field \`body\``) — NOT bound-value content — so unlike the kernel's `Parse`/`Render`
detail (which `error.rs` scrubs for SEC-004) it is safe to surface verbatim here. On a
deserialize error the loader returns before the insert, so **nothing is inserted** (FR-007):
the `?` short-circuits and `insert_and_get` is never reached.

## How SC-003 parity is asserted (the load-bearing V2.2)

`PromptDefinition` derives only `Deserialize, Serialize, Clone, Debug` — **no `PartialEq`**.
So parity is asserted by the **canonical `serde_json::Value` compare**, not `PartialEq` and
not a smoke check:

```rust
fn as_value(def: &PromptDefinition) -> serde_json::Value {
    serde_json::to_value(def).expect("PromptDefinition serializes to a JSON value")
}
// ...
assert_eq!(as_value(from_yaml), as_value(from_json), "SC-003: ...");
```

Re-serializing both parsed defs to `serde_json::Value` and asserting the two `Value`s are
equal is a full structural compare (every field, recursively), and `Value` object comparison
is field-order-insensitive — exactly right since the YAML and JSON source orderings differ.

Asserted over **two** fixtures, not one:
- `single-body.json` vs an equivalent hand-written YAML (string var, `output_model`,
  `metadata`).
- `multi-variant.json` vs an equivalent hand-written YAML — the richer case: two `variants`,
  nested opaque `meta` (float `weight`, `group`, `tags` array), integer var bounds
  (`minimum`/`maximum`), and an `enum` array. Proves parity holds for int/float/array/nested
  shapes, not just the trivial body.

## Norway-safe assertion (V2.5)

`serde_yaml_ng` is backed by `yaml-rust2` (YAML 1.2), where bare `no` / `off` / `yes` are
plain-scalar **strings**, not booleans (the "Norway problem" of YAML 1.1). The carrier is the
free-form `metadata` map (`serde_json::Map<String, Value>` on the generated shape), so the
value lands verbatim as a `serde_json::Value`:

```rust
let yaml = "name: norway\nrole: system\nbody: \"hi\"\nmetadata:\n  country: no\n  toggle: off\n  agree: yes\n";
let def = reg.load_yaml(yaml).expect("...");
assert_eq!(def.metadata.get("country"), Some(&serde_json::Value::String("no".into())));
assert_eq!(def.metadata.get("toggle"),  Some(&serde_json::Value::String("off".into())));
assert_eq!(def.metadata.get("agree"),   Some(&serde_json::Value::String("yes".into())));
```

Each asserts the value is a `Value::String`, never a `Value::Bool` — i.e. NOT coerced to
`false`/`false`/`true` as YAML 1.1 would.

The other tests: V2.1 (`load_yaml` → gettable + fields), V2.3 (constructed object via
`insert` is structurally identical to the loaded one — no second-class path), V2.4 (malformed
YAML, malformed JSON, and shape-violating-but-valid JSON each → `ConsumerError::Load` with
the target name absent from the registry).
