# Critique Report: 019 — pluggable prompt loader

**Date**: 2026-07-08 | **Reviewed**: spec.md, plan.md, tasks.md, contracts, data-model, research,
constitution v2.0.0 (→ v3.0.0), DECISIONS.md, roadmap

## Executive Summary

The highest-stakes spec of the trio: it carries the MAJOR boundary amendment (soften Principle III's
"no I/O") and introduces filesystem I/O. Design is sound and correctly bounded (kernel + construction
stay I/O-free; loader is an opt-in leaf; no container; not fused). **Verdict: ⚠️ PROCEED WITH
UPDATES** — the security review already surfaced MEDIUM hardening (traversal robustness + read cap);
the product/engineering lenses add an adoption concern and two consistency items. No RETHINK.

## Product Lens

- **3a Problem** — Clear: swappable/testable/centralized storage for a real consumer (Bellwether).
- **3b Value / P1 (💡 — half-feature risk):** shipping a loader with NO container (FR-012 defers it)
  risks leaving consumers with exactly the boilerplate #268 wanted to remove — they still write
  `for key in keys: prompts[key] = Prompt.from_yaml(loader.load(key))`. *Assessment:* acceptable for
  v1 (the loader interface + swap/test value stands alone), but the docs (T021) must show the clean
  compose pattern so it doesn't read as half-done, and the deferred-container decision must be
  visible so reviewers know it's intentional. Confirm Bellwether is OK loading N prompts without a
  container.
- **3c Alternatives** — Clear: honest framing (loader value ≠ "couldn't read a file before").
- **3d Edge cases** — Strong: missing key, traversal, not-fused, no-container, distinct errors all
  enumerated.

## Engineering Lens

- **4b/4c Failure & Security (🎯 → see security review):** filesystem I/O with caller-supplied keys.
  The security review (SEC-001 traversal robustness across symlink/TOCTOU/cross-platform; SEC-002
  unbounded-read DoS cap; SEC-003 path/secret scrubbing) is the load-bearing set here. Those MEDIUMs
  should be applied before FileSystemLoader is implemented.
- **4a Architecture (💡):** "per-language ecosystems; loader doesn't cross FFI" (research R5) vs. the
  reality that Python/TS facades may want to reuse the Rust FileSystemLoader. The plan flags this as
  open (R2, lean native). Resolve it: native per-language FileSystemLoader (a few lines of fs code)
  keeps the leaf language-side and avoids marshaling a file's bytes through FFI — but that means the
  traversal guard (SEC-001) must be implemented + tested in EACH language, not once. Make that
  explicit in T004/T008/T009 so the guard isn't Rust-only.
- **4g Dependencies** — Clear: no new standard-package dep; heavy backends deferred.

## Cross-Lens

- **X1 (💡 — amendment completeness):** this softens Principle III AND re-scopes C-08 AND edits the
  roadmap "Never: I/O/storage adapters" list. The tasks (T016-T018) cover all three, but the
  DECISIONS.md entry must be as explicit as spec-015's precedent (which recorded a principle
  redefinition in full). Ensure T018 records the bounded scope (opt-in leaf; kernel/construction
  I/O-free) so a future reader doesn't over-read the softening as "I/O is now fine anywhere."
- **X2 (💡 — sync/async parity):** async TS `load()` vs sync Py/Rust, composed with a sync
  `from_yaml`. The `await loader.load(k)` then sync `Prompt.fromYaml(...)` story is fine but the docs
  must show it so TS users don't trip on the boundary.
